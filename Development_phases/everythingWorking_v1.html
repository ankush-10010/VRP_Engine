VRP Optimization Session Summary

This document is a comprehensive, raw dump of every feature, bug fix, and architectural decision executed during this session to transform the hybrid routing frontend and backend into a polished, production-ready state.

1. Diagnosis of Initial Simulation Constraints

  • Map Rendering: Discovered that the frontend map was using basic <Polyline> components, which drew straight lines from point to point, ignoring actual roads.
  • Algorithm Convergence: Identified that feeding a sparse dataset (orders spread across several days) caused the simulation to easily solve 1-2 orders per hour. This low density resulted in OR-Tools and ALNS finding the exact identical perfect routes immediately, flattening the convergence graph.

2. Map Upgrade Phase 1: Real Road Integration

  • Google Maps API: Upgraded ResultsMap.tsx to instantiate window.google.maps.DirectionsService.
  • Directions Renderer: Replaced the straight <Polyline> with <DirectionsRenderer> to accurately trace physical streets.
  • Custom Markers: Suppressed Google's default markers (suppressMarkers: true) to preserve the custom-designed vibrant UI markers with numbered indexes.

3. Frontend UI: Multi-Select & Color Consistency

  • Multi-Vehicle Selection: Refactored the single activeVehicleId state in ResultsDashboard.tsx into an array activeVehicleIds.
  • Interactive Toggles: Allowed the user to click multiple "v0, v1, v2" pills in the header, or toggle multiple rows in the Fleet Manifest, displaying all selected routes simultaneously on the map.
  • "ALL" Filter: Added a dedicated "ALL" button that instantly clears the array and renders the entire fleet.
  • Consistent Rendering: Fixed a bug where route colors scrambled when filtering. Hardcoded the color assignment in ResultsMap.tsx to use route.vehicle_id % ROUTE_COLORS.length instead of the array index, ensuring Vehicle 0 is always orange regardless of the current view.

4. Backend Overhaul: High-Density Data Preprocessing

  • Old Script Mimicry: Reviewed the old preprocess_single_day.py script and injected its logic directly into the new Modal backend via app/services/parsing.py.
  • Single-Day Compression: Added a FORCE_SINGLE_DAY = True toggle that maps every uploaded order's timestamp to a single base date (Sept 10, 2024), preserving only the hour and minute. This created the massive data density required to force the ALNS and OR-Tools solvers to "fight" over complex routes.
  • Hardcoded Truncation: Capped the parsed data to exactly 50 chronologically sorted orders to prevent Modal timeouts. Left the full-data processing logic perfectly intact but commented out for future toggling.
  • Deployment: Redeployed the modified backend directly to the live Modal container.

5. Map Upgrade Phase 2: Rate Limits & Chunking

Issue Encountered: The heavy data density loaded 25+ stops into single trucks. Google Maps API has a hard limit of 25 waypoints per request, which caused large routes to fail and fall back to straight lines.
  • Waypoint Chunking: Designed a chunking algorithm in ResultsMap.tsx. If a truck has 60 stops, the path is sliced into overlapping segments of 25 stops (1 origin, 1 destination, 23 waypoints).
  • Stitching: Handled the chunks by fetching a DirectionsResult for each segment and rendering a chain of multiple <DirectionsRenderer> components for a single vehicle.
  • OVER_QUERY_LIMIT Fix: Firing multiple chunk requests for 5 trucks simultaneously via Promise.all() triggered Google's client-side rate limit (10 requests/sec). Refactored the promise array into a sequential for...of loop with an injected await new Promise(resolve => setTimeout(resolve, 200)) sleep interval to perfectly pace the API calls.

6. Robust Network Handling: Modal Cold Starts

  • CORS / 404 Network Error: Diagnosed that the frontend occasionally threw Network Errors. This was caused by the Serverless Modal backend taking 5-10 seconds to spin up from a deep sleep ("Cold Start"), causing the initial Axios request to time out or hit a proxy 404.
  • Silent Retry Mechanism: Wrapped the uploadSimulationCsv function in api.ts with a robust 3-attempt retry loop. If a network error occurs, it silently waits 2 seconds and retries. This perfectly masks the cold start, eliminating the error popups entirely.

7. Historical Research: ALNS Hyperparameters

  • Codebase Auditing: Searched the old C:\CODE\Optimization2\final_submission\Codes\hybrid_solver_layers.py file to recover the original ALNS hyperparameters used in the successful legacy project.
  • Recovered Values:
    • Iterations: 50
    • Segment Length: 50
    • Reaction Factor: 0.7
    • Destroy Min: 15% (0.15)
    • Destroy Max: 40% (0.40)

8. Map Upgrade Phase 3: The Ghost Triangle Bug

Issue Encountered: A straight-line triangle connecting the first few stops was permanently glued to the map, sitting directly underneath the newly drawn roads.
  • React State De-sync: Diagnosed that while the DirectionsService was fetching (taking ~1 second), the local state was undefined. The old code treated undefined as a failure and instantly drew a straight-line <Polyline> placeholder.
  • Unmounting Quirk: Discovered a bug in @react-google-maps/api where dynamically swapping a Polyline out for a DirectionsRenderer doesn't cleanly wipe the underlying Google Maps canvas.
  • Resolution: Updated the rendering ternary operator to require an explicitly failed state (null) to draw the Polyline. While fetching (undefined), the component now renders absolutely nothing, completely eradicating the ghost shape.
